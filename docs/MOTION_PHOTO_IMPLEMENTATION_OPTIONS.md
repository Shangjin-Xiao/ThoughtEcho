# Motion Photo 实现方案说明

## 当前实现

当前仓库采用的是 **轻量级 Flutter 方案**：

- 笔记中的图片点开后进入自定义预览页，而不是继续使用 `flutter_quill_extensions` 自带的 `ImageTapWrapper`
- 静态图片预览仍然使用 `PhotoView`，所以缩放、拖拽、全屏查看体验保留
- 仅在 **Android + 本地图片文件** 场景下，尝试识别 Google Motion Photo / Motion Photo JPEG
- 如果识别成功，显示“实况”按钮
- 点击“实况”后会提取图片文件尾部内嵌的视频片段并播放
- 播放结束后自动回到静态照片

### 当前实现的优点

- 改动小，接入快
- 不需要新增原生平台代码
- 不需要引入重型第三方动态照片 SDK
- 保留原有图片预览缩放体验

### 当前实现的限制

- 只覆盖 Android 单文件 Motion Photo
- 依赖 JPEG/XMP/尾部 MP4 的常见格式约定，兼容性不是 100%
- 本质上是“文件解析 + 临时提取播放”，不是系统级媒体能力
- 对未来扩展到 iOS Live Photo、云端资源、相册资源 ID 支持都不够理想

---

## 更好的实现方案

下面按“工程质量”从高到低列出更优路线。

## 方案 A：Android 原生能力 + Platform Channel

### 核心思路

在 Android 侧通过平台通道接入原生实现：

- 使用 `MediaStore` / `ContentResolver` 读取媒体信息
- 读取 XMP / EXIF / motion metadata
- 直接定位 motion video 片段，必要时原生提取
- Flutter 侧只负责展示 UI 和播放

### 优点

- 兼容性最好
- 更容易适配 Google 相册、系统相册、`content://` URI
- 可直接处理系统媒体库，不必要求 Flutter 先拿到绝对文件路径
- 后续支持更多机型差异时更稳

### 缺点

- 开发成本更高
- 需要维护 Android 原生代码
- 测试链路更复杂

### 适用场景

如果要把动态照片作为正式特性长期维护，**这是最推荐的方案**。

---

## 方案 B：专门的 Motion Photo 解析库/插件

### 核心思路

引入专门处理 Motion Photo 的库或自建插件，统一完成：

- motion photo 检测
- metadata 读取
- 视频偏移定位
- 视频提取

### 优点

- 代码量比纯自研原生方案少
- 可以把解析能力封装成单独模块
- Flutter 层逻辑更干净

### 缺点

- 依赖第三方库质量和维护情况
- 若库覆盖不全，最后还是要补原生代码
- 需要额外评估许可证、兼容性和长期维护风险

### 适用场景

适合中期方案：想比当前实现更稳，但又不想马上大规模下沉原生层。

---

## 方案 C：媒体模型升级为“照片资源 + 动态视频资源”

### 核心思路

不要把 Motion Photo 当作“特殊图片文件”，而是把它建模成一个复合媒体对象：

- still image path / uri
- motion video path / uri
- source type（gallery/imported/local/cloud）
- metadata（duration、width、height、mime、capture source）

### 优点

- 架构最清晰
- 未来支持 iOS Live Photo 更自然
- 便于做缓存、导出、备份恢复、跨设备同步
- UI 层只需消费统一的媒体模型

### 缺点

- 需要修改导入链路、数据库、备份恢复、同步逻辑
- 改动范围最大
- 不适合快速上线

### 适用场景

如果未来要把动态照片做成一等公民能力，而不是图片预览的附加功能，**最终应走这条路**。

---

## 方案 D：完全交给系统相册/系统查看器

### 核心思路

点击笔记中的图片后，不自己做动态照片播放，而是跳转系统查看器或外部相册。

### 优点

- 实现最简单
- 能直接继承系统原生体验

### 缺点

- 体验割裂，跳出应用
- 难以控制 UI 和交互
- 对私有目录文件、备份恢复后的路径支持不理想

### 适用场景

只适合作为兜底，不适合作为主实现。

---

## 推荐结论

### 短期推荐

使用当前仓库里已经落地的 **轻量 Flutter 方案**：

- 成本最低
- 已满足“点实况播放一次，结束回静态图”的核心需求
- 对现有笔记图片预览侵入小

### 中长期推荐

如果后续用户真的高频使用动态照片，建议升级到 **方案 A + 方案 C 的组合**：

1. 先用 Android 原生能力提高兼容性
2. 再逐步把动态照片抽象为复合媒体模型

这样后续扩展到：

- iOS Live Photo
- 系统相册 URI
- 云端/同步资源
- 导出与备份恢复

都会更稳、更清晰。
