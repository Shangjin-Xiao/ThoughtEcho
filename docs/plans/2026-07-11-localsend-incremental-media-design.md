# LocalSend 增量媒体同步设计

## 目标

LocalSend 同步继续发送完整的结构化笔记数据并使用现有 LWW 合并，但媒体文件只发送接收端缺失或大小不同的部分。不传播媒体删除，不修改数据库 schema，并兼容尚未支持媒体清单的旧版本。

## 协议与数据流

发送端仍先调用 ThoughtEcho 的 `sync-intent` 审批接口。接收端只有在用户批准后才扫描应用文档目录下的 `media/`，并在审批响应中附带版本化媒体清单。清单键为使用 `/` 分隔的安全相对路径，值为文件字节数；不返回绝对路径。旧版本响应不含清单，发送端据此回退到当前全量媒体备份。

发送端扫描备份服务已经识别为有效引用的媒体，根据接收端清单筛选：路径不存在或大小不同则写入同步 ZIP，路径和大小相同则跳过。结构化数据始终完整导出，且媒体路径仍转换为可跨设备恢复的 `media/...` 相对路径。接收端沿用现有备份解压、媒体恢复和 LWW 合并流程，因此无需增加第二种接收格式。

## 安全与失败处理

媒体清单只在同步请求获批后返回，避免新增可匿名枚举本地文件的接口。扫描只接受 `media/` 下的普通文件并规范化相对路径。任何清单解析错误、未知版本或旧端缺少能力字段都触发全量媒体回退，不能因为协商失败漏传附件。

媒体删除不随同步传播。若接收端存在发送端没有的文件，它会保留；清理由现有媒体引用/清理能力独立负责。同步取消继续贯穿审批、打包和发送，失败时不推进任何额外状态，因为本方案不引入持久化同步游标。

## 验收标准

- 接收端审批响应可返回规范化的媒体大小清单。
- 相同路径且相同大小的媒体不进入同步 ZIP。
- 缺失或大小不同的媒体进入同步 ZIP。
- 不支持新清单的目标设备仍收到完整媒体备份。
- 完整笔记数据、LWW 合并、取消和旧版 LocalSend 行为保持可用。
