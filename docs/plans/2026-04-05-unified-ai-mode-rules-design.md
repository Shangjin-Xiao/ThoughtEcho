# 统一 AI 容器模式、命令与权限规则说明 (2026-04-05)

> 本文档用于补充并覆盖近期 `AIAssistantPage` / `ExplorePage` / Agent / 工作流相关计划中的歧义点。
> 如与旧设计稿、旧实施计划、旧进度记录冲突，以本文档为准。

## 目标

统一 `AIAssistantPage` 的入口、模式切换、slash 命令、Agent 权限边界，以及旧 AI 工作流的继承方式，供后续实现 agent 直接照此落地。

## 实现标准

本次实现目标不是“最小正确改动”，而是面向后续长期维护的 SOTA 级重构。

要求：

- 优先建立清晰、可扩展、可维护的模式与工作流架构。
- 允许进行必要的结构性重组，只要最终行为完全符合本文档。
- 不接受靠大量分支判断和临时兼容拼出来的脆弱实现。
- 不要求为了少改几行代码而保留明显错误的页面职责划分。
- 允许把现有硬编码 slash 分支重构为统一命令/工作流注册表。
- 允许把现有普通对话、笔记对话、Agent 三套逻辑重新收敛为统一状态机或统一页面控制层。

但同时要求：

- 尽量复用现有 prompt、分析参数、分析风格、数据服务与已有业务逻辑。
- 不要为了“重构得更漂亮”而擅自改变用户已有功能语义。
- 不要删除旧能力，除非本文档明确允许。

## 核心原则

1. `AIAssistantPage` 是唯一 AI 入口容器。
2. 模式切换发生在同一页面内，不通过 slash 命令切换，不通过跳转其他页面切换。
3. slash 命令不是切模式，也不是新建会话，只是显式触发某个工作流。
4. 所有旧 AI 工作流能力都要保留，优先复用原 prompt、原参数、原业务逻辑。
5. 工作流参数卡片和结果卡片必须内嵌在 `AIAssistantPage` 会话流中，不能离开 AI 页。

## 页面入口与模式

`AIAssistantPage` 中存在三种实际工作状态：

1. 普通对话模式（General Chat）
2. 笔记绑定对话模式（Note-bound Chat）
3. Agent 模式（Agent）

### Explore 入口

从 `ExplorePage` 进入 AI 时：

- 默认进入普通对话模式。
- 页面内允许切换到 Agent 模式。
- 需要单独记住 `Explore` 入口的最后模式选择。
- 动态引导语应显示在 `AIAssistantPage` 内，而不是 `ExplorePage` 内。

### 笔记入口

从单条笔记、问笔记按钮、编辑器 AI 功能进入 AI 时：

- 默认进入笔记绑定对话模式。
- 页面内允许切换到 Agent 模式。
- 切换到 Agent 模式后，当前笔记绑定上下文保留，不重开会话。
- 需要单独记住“笔记入口”的最后模式选择。

### 模式记忆

模式记忆需要分开保存：

- `Explore` 入口记住自己的最后模式。
- 笔记入口记住自己的最后模式。
- 两者不能互相污染。

## UI 要求

### 模式切换

- `对话` / `Agent` 切换按钮放在输入框上方。
- 这组按钮属于页面级模式开关，不属于 slash 菜单。
- 切换模式不离开页面，不清空当前会话 UI。

### 输入框

- 默认提示文案应明确告诉用户：可以直接对话，也可以输入 `/` 查看功能。
- 推荐语义：`输入 / 查看功能，或直接对话`。

### 动态引导语

- 从 `Explore` 进入 AI 时，在 `AIAssistantPage` 顶部或首条消息处显示动态引导语。
- 这段引导语来源于 Explore 的数据概览总结逻辑。
- 这段引导语用于开场和引导，不强制作为后续模型历史上下文的一部分。

## slash 命令规则

### 定义

slash 命令的本质是“显式触发工作流”。

- 不是切换模式。
- 不是创建新对话。
- 不是唯一入口。
- 更像是为当前消息追加一段强提示或明确调用某条工作流。

### Agent 模式下的行为

在 Agent 模式下：

- 即使用户没有输入 slash，AI 也可以通过自然语言完成原本需要 slash 才能触发的工作。
- 这些能力应以 Agent skills / tools 的形式加载到 Agent 模式中。
- 换言之，slash 在 Agent 下是“显式触发方式”，不是“唯一触发方式”。

### 普通模式下的行为

在普通对话模式下：

- slash 依然用于显式触发工作流。
- 如果某项工作流需要额外权限或需要具体笔记，但当前上下文不满足，则仍显示该命令，但执行时给出明确提示，不静默失败。

## 权限模型

### 普通对话模式（从 Explore 进入）

能力：

- 可以与 AI 做普通对话。
- AI 不能自行访问其他笔记。
- AI 不能自行搜索全库笔记。

例外：

- 当用户显式触发 `/智能洞察` 时，程序应像旧实现一样，主动将“所有笔记”发送给 AI 做分析。
- 这是“程序托管的全量分析工作流”，不是 AI 自主搜索。

如果用户在该模式下触发如下需要具体笔记的命令：

- `/润色`
- `/续写`
- `/深度分析`
- `/分析来源`

则系统/AI 应提示：

- 请先关联一条具体笔记，或
- 切换到 Agent 模式后说明要处理哪一条笔记

### 笔记绑定对话模式

能力：

- 可以围绕当前绑定笔记进行普通对话。
- 可以回答、解释、总结、讨论当前笔记。
- AI 不能自行搜索其他笔记。

工作流行为：

- `/润色`、`/续写`、`/深度分析`、`/分析来源` 基于当前笔记执行。
- `/智能洞察` 仍然沿用旧逻辑：程序主动发送“所有笔记”给 AI 分析。
- 这里的 `/智能洞察` 不是只分析当前笔记，也不是 AI 自主搜索。

限制：

- 在笔记绑定的普通对话模式下，如果用户没有显式输入 slash，AI 不应进入可应用编辑结果流。
- 也就是说，普通聊天可以讨论“如何润色”，但不能直接产出带“替换原笔记 / 追加到末尾”按钮的可应用结果卡片。
- 只有用户显式输入对应 slash 命令后，才允许调用对应编辑类工作流并产出可应用结果。

### Agent 模式

能力：

- 可以使用 Agent skills / tools。
- 可以自行搜索笔记、读取笔记、继续探索。
- 如果已有绑定笔记，则保留该笔记作为强上下文。
- 如果没有绑定笔记，则 AI 不应假设目标笔记，仍需用户说明是哪一条，再由 Agent 搜索定位。

补充说明：

- Agent 模式不需要每次进入都自动恢复历史对话。
- 默认可以新开会话。
- 是否查看历史会话由用户主动操作。

## 命令展示策略

所有命令都应展示，不做完全隐藏。

采用“始终展示，但根据当前模式和上下文给出不同可执行提示”的策略。

例如：

- 可直接执行
- 需要关联笔记
- 建议切换到 Agent

## 必须保留的旧功能

不能只保留 `/润色` 与 `/续写`。以下能力必须继承：

1. `/润色`
2. `/续写`
3. `/深度分析`
4. `/分析来源`
5. `/智能洞察`
6. 原有“分析所有笔记”工作流
7. 原有智能洞察中的分析项
8. 原有智能洞察中的分析风格选项

### 智能洞察的继承要求

这里的“智能洞察”特指原应用中“分析所有笔记”的那套工作流。

必须尽量沿用原有：

- prompt
- 参数结构
- 分析项
- 风格选项
- 业务处理逻辑

尤其要保留原有可选项，例如：

- 全面分析
- 情感洞察
- 其他原本已存在的分析项

以及原有分析风格，例如：

- 专业
- 友好
- 风趣
- 其他原本已存在的风格

## 工作流卡片要求

旧能力如果原本有参数选择卡片、工作流卡片、结果卡片：

- 可以继续复用这些卡片 UI 和配置项。
- 但这些卡片必须内嵌到 `AIAssistantPage` 消息流中。
- 不能因为触发 `/智能洞察`、`/深度分析`、`/分析来源` 而跳去旧页面。
- 不能退出 AI 页后再回填。

### 结果应用闭环

对于 `/润色`、`/续写` 等可编辑结果：

- 继续使用结果卡片展示。
- 支持 `替换原笔记` / `追加到末尾`。
- 这个闭环只应在满足权限条件时出现。

## 改动点清单

实现 agent 应把以下改动点在代码中明确落地，而不是只修表面行为。

### 1. AI 页面入口模型

- 为 `AIAssistantPage` 建立明确的入口来源枚举，例如：`explore`、`note`。
- 为 `AIAssistantPage` 建立明确的页面模式枚举，例如：`chat`、`noteChat`、`agent`，或其他等价但清晰的设计。
- 入口来源与当前模式必须同时存在，不能继续只靠 `quote != null` 这种隐式推断撑完整逻辑。

### 2. 模式持久化

- 增加持久化字段，分别保存：
  - `Explore` 入口最后模式
  - 笔记入口最后模式
- 模式切换时立即更新持久化状态。
- 页面初始化时按入口来源读取对应的最后模式。

### 3. AIAssistantPage 页面结构重组

- 在输入框上方加入 `对话` / `Agent` 模式切换区。
- 将动态引导语整合进 `AIAssistantPage`。
- 将 slash 菜单、参数卡片、结果卡片统一纳入会话流。
- 页面应具备承载“普通消息 + 配置卡片 + 结果卡片 + Agent 状态”的统一消息模型或等价机制。

### 4. slash 命令架构重组

- 不要继续把 slash 逻辑分散为多个 `if (text == '/xxx')`。
- 建议建立统一命令描述层，至少包含：
  - 命令名
  - 展示名
  - 是否需要绑定笔记
  - 是否允许在普通模式下执行
  - 是否允许在 Agent 中自然语言隐式触发
  - 是否可能产出可应用编辑结果
  - 对应旧 prompt / 工作流入口

### 5. 旧工作流内嵌化

- 将旧的 `/智能洞察`、`/深度分析`、`/分析来源`、相关参数选择与结果展示迁移到 `AIAssistantPage` 内。
- 可以复用原页面中的参数组件、卡片、prompt 组装逻辑。
- 但最终交互必须发生在 AI 页内部，不能通过页面跳转完成。

### 6. 普通模式与 Agent 模式权限分层

- 普通模式不能自行检索或探索其他笔记。
- Agent 模式可以检索和探索其他笔记。
- `/智能洞察` 在普通模式和笔记模式下，仍走“程序主动发送所有笔记”的托管流程。
- Agent 模式下可以同时拥有：
  - 程序托管型工作流
  - AI 自主搜索型能力

### 7. 编辑闭环治理

- `SmartResultCard` 一类可应用结果只应在满足权限时产生。
- 笔记普通对话模式下，未显式 slash 时不能直接产出可编辑结果。
- 笔记普通对话模式下，显式 `/润色`、`/续写` 时可进入可应用闭环。

### 8. Explore 与旧入口同步

- `ExplorePage` 的 AI 入口要按新入口模型接入 `AIAssistantPage`。
- 所有旧“问笔记”入口要按笔记入口模型接入 `AIAssistantPage`。
- 旧 wrapper 页面可以保留，但行为必须与新规格一致。

### 9. 旧 prompt / 参数复用

- 找出当前项目中与以下能力相关的旧 prompt、旧参数、旧风格：
  - 智能洞察
  - 深度分析
  - 分析来源
  - 润色
  - 续写
- 不要新造一套语义接近但行为不同的 prompt 替代品。

### 10. 验证要求

- 至少补足与本次重构直接相关的分析和页面级测试。
- 验证重点不是“有没有页面”，而是：
  - 不同入口的默认模式是否正确
  - 模式是否分别记忆
  - slash 是否只触发工作流
  - 普通模式权限是否正确受限
  - Agent 是否具备自然语言触发能力
  - 旧工作流是否仍然能在 AI 页内完成

## 推荐实现矩阵

### Explore 进入 + 普通对话

- 普通聊天：允许
- `/智能洞察`：允许，程序发送所有笔记
- `/润色`：显示，但提示需关联笔记
- `/续写`：显示，但提示需关联笔记
- `/深度分析`：显示，但提示需关联笔记
- `/分析来源`：显示，但提示需关联笔记
- AI 自主搜索其他笔记：不允许

### 笔记进入 + 普通对话

- 普通聊天：允许，仅围绕当前笔记
- `/智能洞察`：允许，程序发送所有笔记
- `/润色`：允许
- `/续写`：允许
- `/深度分析`：允许
- `/分析来源`：允许
- 不输 slash 时直接进入可应用编辑：不允许
- AI 自主搜索其他笔记：不允许

### Agent 模式

- 自然语言触发能力：允许
- 不输 slash 也可执行复杂任务：允许
- 搜索/探索其他笔记：允许
- `/智能洞察`：仍可保留为显式程序托管工作流
- 无绑定笔记时做润色/分析来源：先要求用户说明目标，再搜索定位

## 对实现 agent 的明确要求

1. 不要把 slash 命令实现成“模式切换器”。
2. 不要把 slash 命令实现成“新对话入口”。
3. 不要把旧分析能力删成只剩 `/润色` 与 `/续写`。
4. 不要把旧工作流重新拆成多个独立页面跳转。
5. 不要让普通模式偷偷拥有 Agent 的全库探索权限。
6. 不要让 Explore 进入的普通模式默认访问其他笔记。
7. 不要让笔记普通对话在未显式 slash 时直接产出可应用编辑卡片。

## 建议的落地方向

1. 在 `AIAssistantPage` 增加统一模式枚举与入口来源枚举。
2. 用持久化设置分别保存 `Explore` 入口与笔记入口的最后模式。
3. 将 slash 命令抽象为“工作流描述 + 权限要求 + 是否需要绑定笔记 + 是否允许产出可应用结果”。
4. 将旧 `/智能洞察`、`/深度分析`、`/分析来源` 参数卡片迁移为会话内卡片。
5. Agent 模式加载 skills/tools；普通模式仅允许受限工作流。
