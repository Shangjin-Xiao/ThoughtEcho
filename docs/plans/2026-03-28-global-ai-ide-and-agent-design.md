# Explore 页中的统一 AI 助手交互设计

> **Goal:** 只定义统一 AI 容器的产品交互，不再重复数据库迁移和地图实现细节。这里回答的是“用户如何感知和使用统一 AI”，不是“底层怎么建表”。

## 1. 文档边界

本方案负责：

- Explore 中统一 AI 容器的入口和布局角色
- 全局上下文与单笔记上下文的交互规则
- Chat / Agent 双模式
- `/` 命令工作流
- 结果展示与“一键应用”体验
- 旧入口迁移策略

本方案不负责：

- `chat_sessions` / `chat_messages` 的表结构和迁移
- Agent runtime 的工具循环实现细节
- `poiName`、地图选点、地图回忆的具体页面实现

实现细节分别参考：

- `2026-03-26-ai-chat-history-and-agent-design.md`
- `2026-03-26-map-location-picker-and-note-map.md`
- `2026-03-28-master-refactoring-explore-page-and-ai-ide.md`

## 2. 统一 AI 的产品定义

ThoughtEcho 不再维护多个平行的 AI 页面，而是只保留一个统一 AI 容器。

- 从 `Explore` 进入时，它是一个**全局 AI 助手**。
- 从单条笔记或编辑器进入时，它是一个**带当前笔记上下文的同一容器**。

也就是说，用户面对的不是“不同 AI 功能页”，而是“同一个会话容器在不同上下文下工作”。

## 3. 入口规则

### 入口 A：从 Explore 进入

- 默认无绑定笔记。
- 用户可以直接自由提问。
- 用户也可以主动添加一篇或多篇笔记作为上下文。
- 这条路径更接近“全局搜索 / 总结 / 洞察 / Agent 探索”。

### 入口 B：从单条笔记或编辑器进入

- 自动挂载当前笔记作为首要上下文。
- 默认更适合问答、润色、续写、深度分析等围绕当前笔记展开的任务。
- 如果后续用户手动再附加其他笔记，上下文可以扩展，但“当前笔记”仍是主上下文。

## 4. 模式切换

统一容器提供两种模式，但它们共享同一个会话壳层：

### Chat Mode

- 默认模式。
- 轻量、快速。
- 不主动调用工具。
- 适合普通问答、局部润色、快速解释、总结。

### Agent Mode

- 适合“需要检索、对比、汇总、跨笔记推断”的任务。
- 允许模型触发工具，例如搜索笔记、拉取指定范围的数据、校验洞察来源。
- MVP 阶段优先保证稳定性，不强求完整流式工具过程。

## 5. `/` 命令的真正角色

`/` 命令不是为了再造一组“新页面”，而是为了把原本分散的 AI 能力变成统一容器内的工作流入口。

推荐首批命令：

- `/智能洞察`
- `/周期报告`
- `/深度分析`
- `/润色`
- `/续写`

设计原则：

- 命令触发的是**会话中的结构化卡片或工作流状态**，不是跳去另一个页面。
- 每个命令都应能在当前会话里留下可追问的结果。
- 当命令依赖“当前笔记”时，如果上下文缺失，要明确提示用户先绑定笔记，而不是静默失败。

## 6. 结果渲染规则

统一 AI 容器里至少存在两类特殊消息：

### 洞察 / 报告类结果

- 以可追问的结果卡片或富文本块展示。
- 可附带“保存到洞察记录”或其他持久化动作。
- 保存动作是结果的后处理，不应把用户踢出当前会话。

### 文本生成类结果

- 以 `artifact` 风格的文本块展示。
- 提供明确的 **[应用到编辑器]** 操作。
- 点击后进入对应编辑器，并执行替换、插入或追加。

## 7. 非目标

以下内容不应该继续发生：

- 给 `AIFeaturesPage` 再扩第三个 Tab
- 为润色、续写、深度分析分别维护长期独立页面
- 让“问笔记”和“全局 AI”成为两个长期并存但能力重复的容器
- 在这个文档里继续展开 schema 或地图实现细节

## 8. 迁移策略

### 第一阶段

- 保留旧入口，但让旧入口优先跳转到统一容器或兼容 wrapper。
- 用户不感知“系统内部是否已切到新架构”。

### 第二阶段

- Explore 成为底部导航中的正式主入口。
- 原 `AIFeaturesPage` 只保留过渡职责，不再承接新功能。

### 第三阶段

- 等 Explore 和统一 AI 容器稳定后，再清理旧问答页、独立分析页和散落按钮。
