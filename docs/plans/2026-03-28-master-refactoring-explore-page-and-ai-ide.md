# 探索页 / 统一 AI / 地图回忆 总计划

> **Goal:** 把当前分散的入口收敛成一个清晰的产品心智。`Explore` 负责三件事：看数据、和 AI 交互、从空间维度回顾笔记。其余计划文档只负责各自子系统的细化实现，不再重复讲整体故事。

## 1. 澄清后的需求主线

这几份未跟踪文档其实围绕的是同一个重构主题，只是层级混在了一起：

1. **入口重构**
   - 底部导航里的 AI 入口不应继续演化成更多独立页面，而应升级为统一的 `Explore`。
   - `Explore` 是用户理解 AI、数据概览、地图回忆的总入口。

2. **统一 AI 容器**
   - “问笔记”“智能洞察”“润色/续写/深度分析”本质上都应收敛为同一个会话容器。
   - 区别不是“不同页面”，而是“同一容器在不同上下文和命令下触发不同工作流”。

3. **地图能力**
   - 地图选点先解决“给单条笔记补充更精确的位置语义”。
   - 地图回忆再利用这些结构化位置信息，作为 `Explore` 里的空间探索能力。

4. **底层基建**
   - AI 聊天持久化和 Agent 运行时是统一 AI 容器的基础。
   - `poiName` 和地图选点是地图回忆的基础。
   - 两者共享同一轮 schema 协调，因此不能各写各的迁移编号。

5. **不在本计划范围**
   - 回收站方案单独处理，不并入本重构。

## 3. 统一后的产品结构

`Explore` 页面只保留三块：

1. **顶部：数据总览**
   - 延续现有数据概览能力，作为进入探索的第一屏。

2. **中部：统一 AI 助手**
   - 这是所有 AI 能力的公共入口。
   - 从 Explore 进入时默认是“全局上下文”。
   - 从单条笔记或编辑器进入时，是“绑定当前笔记上下文”的同一个容器。

3. **底部：地图回忆入口**
   - 只是入口卡片，不在 Explore 首屏内直接铺完整地图。
   - 点进去之后进入独立的全屏地图回忆视图。

## 4. 依赖关系

实施上应当按依赖拆分，而不是按“页面名字”拆分：

1. **第一层：数据基础**
   - 聊天表、消息模型、会话模型
   - `Quote.poiName`
   - schema 版本统一协调

2. **第二层：单点能力**
   - Note QA 持久化
   - Agent 工具运行时
   - 地图选点页

3. **第三层：统一壳层**
   - Explore 页面
   - 统一 AI 容器
   - 旧入口 wrapper 跳转

4. **第四层：高阶体验**
   - `/` 命令工作流
   - 地图回忆视图
   - 一键应用到编辑器

## 5. 推荐实施顺序

### Phase A: 基建先落地
- 完成聊天持久化与多轮上下文。
- 完成 `poiName` 字段和地图选点页。
- 明确最终 schema 版本顺序，避免多个计划都写死 `v20`。

### Phase B: 统一 AI 容器成型
- 先让现有问笔记入口切到统一容器或兼容 wrapper。
- 再加上 Chat / Agent 模式切换。
- 最后把 `/智能洞察`、`/润色`、`/续写`、`/深度分析` 收拢成命令式工作流。

### Phase C: Explore 接管主入口
- 新建 Explore 页承接数据总览、统一 AI 入口、地图入口。
- 旧 `AIFeaturesPage` 不再扩新结构，只做过渡跳转。

### Phase D: 地图回忆补完
- 在 Explore 底部增加地图入口卡片。
- 完成 marker / cluster / 文本摘要卡片的地图回忆视图。

## 6. 成功标准

- 用户只需要理解一个概念：去 `Explore`，而不是在多个 AI 页面里找入口。
- AI 能力按“上下文 + 命令”组织，而不是按“页面”组织。
- 地图选点和地图回忆被视为同一条产品线的前后两段，而不是两个孤立功能。
- 计划文档之间只保留引用关系，不再大段重复。
